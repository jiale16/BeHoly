> 本文档为 **增量需求**，仅做产品定义与设计规划，**不含实现代码**。
> 适用范围：BeHoly（个人离线屏幕内容安全监测 App）命中提醒后的「反思引导」能力。

## 0. 背景与工程约束（设计边界）

以下事实来自当前工程代码，本次设计**必须基于**它们，不得凭空假设：

| 现状对象 | 关键事实 | 对本次设计的影响 |
|---|---|---|
| `RepentanceActivity`（`com.example.beholy.ui`） | `btn_repent`（"我愿意悔改归向神"）与 `btn_close`（"返回 BeHoly"）**均只调用 `finish()`**，无任何后续引导 | 本次需新增一个结构化反思表单；按钮行为需要分流 |
| `ScreenMonitorService.showRepentance(reason: String)` | 通过 `FullScreenIntent` 启动 `RepentanceActivity`，但 **Intent 未携带 `reason` 与命中时间戳**；`onHit(result)` 处虽有 `result.timestamp` 可用却未传出 | 需新增 extra 透传命中类型与时间给界面 |
| `DetectionResult`（`com.example.beholy.data`） | 字段 `isImageNsfw:Boolean`、`imageScore:Float`、`isTextHit:Boolean`、`hitWords:List<String>`、`recognizedText:String`、`timestamp:Long` | 反思表单的「检测类型/时间」数据源 |
| `HitLogger`（`com.example.beholy.util`） | 向私有文件 `detection_log.txt` 追加 `时间 \| 类型 \| 详情`；有 `hitCount()` 但**无"上次命中/反思时间"查询方法** | 反思记录不应复用该纯文本日志；建议新建独立的 JSON 存储 |
| `Constants`（`com.example.beholy.data`） | 集中常量（阈值、通知 ID、模型/词库文件名、224 输入尺寸、`LOG_TAG="BeHoly"` 等） | 新增 extra key 常量建议在此统一登记 |
| 技术栈 | Kotlin + AndroidX；compileSdk/targetSdk/minSdk = 34/34/29；AGP 8.2.2；Gradle 8.3；JDK 17；UI 现用 `findViewById`（非 viewBinding） | 新增界面沿用 `findViewById` 风格、降低编译风险 |
| **强制离线** | Manifest **不声明 INTERNET**，任何代码**不得有网络调用** | 反思数据仅存本地；不做云端同步/上传 |

## 1. 产品目标

当前 BeHoly 在命中成人内容时，只会弹出「悔改归向神」的提醒界面，用户点击「我愿意悔改归向神」后即直接关闭——提醒到此为止，但**没有帮助用户在当下停下来、看清自己、并思考如何避免下一次**。这导致每一次命中都停留在"被提醒"的瞬间，难以沉淀为真正的改变。

「悔改反思日志」要做的，是把那一次次的"被提醒"转化为**主动的反思与预防**：命中后若用户愿意悔改，引导其用结构化的方式记录本次浏览的感受、途径、反思与避免方案，并自动呈现"距上次反思多久"。这些记录仅保留在设备本地、完全离线，构成一个**只属于用户自己、可随时回顾的成长轨迹**——让悔改不只是一次情绪，而是一段看得见、能复盘、可被鼓励的生命记录。

## 2. 用户故事

1. **作为**被提醒的用户，**我**在命中后点击「我愿意悔改归向神」，**希望**进入一个温柔的反思表单填写本次心情与浏览方法，**以便**把当下的感受记下来、日后能回顾自己是怎么走过来的。
2. **作为**反复挣扎的用户，**我**在表单里看到「距上次反思 3 天 2 小时」，**希望**这个间隔被自动算好，**以便**直观地感知自己的状态是在变好还是反复。
3. **作为**想要改变的用户，**我**能在「以后的避免方案」里写下具体计划（如"睡前把手机放客厅"），**希望**它能被保存，**以便**下次软弱时回看自己的承诺。
4. **作为**时间有限/情绪低落的的用户，**我**可以选择「跳过」而不必勉强填写，**希望**系统尊重我的选择、不强行要求，**以便**在状态差时也不会因"必须填"而产生抵触。
5. **（P2 期望）作为**坚持了一段时间的用户，**我**能打开历史记录看到过往反思列表与连续未命中天数，**希望**得到金句鼓励，**以便**获得持续走下去的动力。

## 3. 需求池（P0 必须 / P1 建议 / P2 可选）

> 优先级含义：**P0** = 必须（本迭代交付）；**P1** = 建议（体验关键，尽量同批）；**P2** = 可选（后续迭代）。
> 每条含「编号 / 描述 / 优先级 / 验收标准」。

### 3.1 P0（必须）

| 编号 | 描述 | 优先级 | 验收标准 |
|---|---|---|---|
| **P0-1** | 点击 `btn_repent` 后**不再直接 `finish()`**，而是启动结构化反思表单 `RepentanceFormActivity`（`com.example.beholy.ui`）。`btn_close` 保持原行为（仅关闭）。 | P0 | 点击"我愿意悔改归向神"→ 进入反思表单；点击"返回 BeHoly"→ 仅关闭界面，不进表单（默认，详见待确认项 Q2）。 |
| **P0-2** | 反思表单至少包含以下字段：①**感受心情**（预设 chips：空虚/愧疚/焦虑/愤怒/麻木/平静，可多选 + 可选文字补充）；②**看的方法**（预设 chips：浏览器/短视频/社交软件/论坛/相册/其他 + 可选说明）；③**反思**（多行文本）；④**以后的避免方案**（多行文本）；⑤**距离上次反思时间**（系统自动计算并显示，如"距上次 3 天 2 小时"）。 | P0 | 五个字段均出现且可交互；心情/方法支持多选与补充；多行文本可输入；距离字段非用户手填、由系统算出。 |
| **P0-3** | 「保存」将本次反思写入**本地持久化存储**（推荐：轻量 JSON 文件，沿用 `HitLogger` 的"私有 `filesDir` 追加写入"风格，降低编译风险；JSONL 一行一条记录最易追加/解析）。 | P0 | 保存后，在同设备 `filesDir` 下生成/追加反思记录文件；记录包含命中时间、类型、心情、方法、反思、避免方案；**无任何网络写入**。 |
| **P0-4** | 「距离上次反思时间」在**打开表单时即计算**：读取上一条已保存反思记录的时间戳，与"本次命中时间"之差。 | P0 | 表单首屏即显示间隔；若无历史记录则显示"首次反思"之类的友好占位；差值实时由本地计算。 |
| **P0-5** | 表单提供「**保存**」与「**跳过**」两个出口。**跳过 = 不写任何反思内容**（明确取舍：P0 阶段不记录"已悔改未填"标记，保持记录纯净、尊重用户选择；如需为 P2 统计服务，后续再以独立开关追加）。 | P0 | 点击「保存」→ 写入记录并返回；点击「跳过」→ 直接关闭、**不生成**反思记录文件条目。 |
| **P0-6** | `ScreenMonitorService.showRepentance` 需把 `reason` 与命中时间戳透传给界面；`RepentanceActivity` 接收后，在启动 `RepentanceFormActivity` 时一并转发这两个 extra。 | P0 | `RepentanceFormActivity` 能拿到本次 `reason`（如"成人画面"/"敏感文字"）与 `hitTime`（= `DetectionResult.timestamp`）；全程无新增权限、无网络。 |

**反思记录数据模型（设计级，非代码）**，每条 JSON 对象建议字段：

```
{
  "hitTime": Long,          // 检测命中时间戳（= EXTRA_HIT_TIME，作为"距上次"基准）
  "reason": String,         // "成人画面" / "敏感文字" / "成人画面、敏感文字"
  "moods": [String],        // 选中的心情 chips
  "moodNote": String,       // 心情补充（可选）
  "methods": [String],      // 选中的"看的方法" chips
  "methodNote": String,     // 方法说明（可选）
  "reflection": String,     // 反思文本
  "avoidancePlan": String,  // 避免方案文本
  "gapSinceLastMs": Long    // 距上一条反思记录的间隔（打开时算出，便于回看）
}
```

> 存储文件名建议：`reflection_log.json`（JSONL，一行一条），与 `detection_log.txt` 同目录（`context.filesDir`）。

### 3.2 P1（建议）

| 编号 | 描述 | 优先级 | 验收标准 |
|---|---|---|---|
| **P1-1** | 表单顶部展示**本次检测类型**（成人画面/敏感文字）与**发生时间**（由 `EXTRA_REASON` + `EXTRA_HIT_TIME` 渲染）。 | P1 | 表单可见"类型：成人画面 ｜ 发生：yyyy-MM-dd HH:mm"之类信息。 |

| **P1-3** | 中文文案**统一且语气温柔引导而非指责**（如"此刻你愿意停下来看看自己吗？"），全表单术语一致。 | P1 | 走查文案无指责性措辞，与 App 整体悔改呼召基调一致。 |

### 3.3 P2（可选）

| 编号 | 描述 | 优先级 | 验收标准 |
|---|---|---|---|
| **P2-1** | **历史记录查看页**：列表展示过往反思（时间 / 心情 / 方法 / 摘要）。 | P2 | 可打开列表、按时间倒序、点击查看单条详情。 |

## 4. UI 设计稿（文字线框 · `RepentanceFormActivity`）

### 4.1 需要接收的 Intent Extra

| Extra Key（建议常量名） | 类型 | 来源 | 说明 |
|---|---|---|---|
| `EXTRA_REASON` | `String` | `ScreenMonitorService.showRepentance(reason)` → `RepentanceActivity` 转发 | 如 "成人画面" / "敏感文字" |
| `EXTRA_HIT_TIME` | `Long` | `DetectionResult.timestamp`（`onHit(result)` 处已有） | 用于"距上次"基准与时间显示 |

> Extra key 常量建议在 `Constants.kt` 或各 Activity 的 `companion object` 中集中登记（沿用现有 `EXTRA_RESULT_CODE`/`EXTRA_DATA` 风格）。

### 4.2 字段布局（首屏自上而下）

```

│ ［首次反思］（无历史记录时占位）                │
├─────────────────────────────────────────────┤  ← P1-1（建议）
```

## 5. 待确认问题（需用户/主理人拍板）

| 编号 | 问题 | 默认建议 |
|---|---|---|
| **Q1** | 持久化用 **JSON 文件** 还是 **Room**？ | **推荐 JSON 文件**（JSONL）。理由：沿用 `HitLogger` 私有 `filesDir` 追加风格、零额外依赖、无 schema 迁移、编译风险低、完全离线；仅当 P2 需要复杂查询/检索时再考虑 Room。 |
| **Q2** | 点击 `btn_close`（"返回 BeHoly"）是否也进表单？ | **默认仅 `btn_repent` 进表单**；`btn_close` 保持仅关闭。理由：尊重"仅想退出"的用户意图，避免强迫。 |
| **Q3** | "距离上次"的基准：距上次**保存的反思记录**，还是距上次**命中**？ | **默认距上次"保存的反思记录"**。理由：反思日志的间隔应以"用户实际完成的反思"为刻度，更能反映成长节奏。 |
| **Q4** | 是否需要**历史记录查看页**（P2-1）？ | 本迭代先不做；若确认需要，排入后续迭代（影响是否现在就设计列表 UI 与入口）。 |
| **Q5** | 心情/方法**预设选项是否需可自定义扩充**（如用户自建标签）？ | P0 先用固定预设；自定义扩充列入 P2，避免首版复杂度。 |
| **Q6** | 新增 extra 常量放 `Constants.kt` 还是 Activity `companion object`？ | 推荐放 `Constants.kt` 集中管理（与现有常量风格一致），或至少在发送/接收两端共用同一常量。 |
| **Q7** | 「跳过」是否记极简标记（"已悔改未填"）？ | **P0 不记**（见 P0-5 取舍）；若后续 P2 统计需要，再以独立开关追加，不阻塞首版。 |
